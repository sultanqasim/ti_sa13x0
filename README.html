SA1350 System Overview, GUI Version 1.3

Software Overview

This software is part of the SA1350 Project which implements an RF spectrum analyzer based on the TI CC13x0 SimpleLink RF SOC. The SA1350 is capable of performing RF measurements over three frequency bands: 2152MHz to 2635MHz, 861MHz to 1054MHz, and 431MHz to 527MHz. RF measurement parameters can be adjusted through user input and RF measurement results viewable in a spectrum plot from within a graphical user interface (GUI).

Software Implementation

The SA1350 Software is based on a Qt GUI for configuring and displaying the RF spectrum and a Windows DLL for interfacing between the GUI and the CC13x0 LaunchPad hardware.

Windows DLL

The Windows DLL is based upon a CThread Worker Thread wrapper class which enables a SA1350 driver that manages the USB’s device list, initialization, status, and communication.

All Worker Thread related files which provide the low- and mid-level layer of the DLL begin with a lowercase c. cMutex.cpp, cEvent.cpp, cDriver.cpp, and cThread.cpp contain the Worker Thread implementation which provides the framework and various classes utilized by the mid-level frameworks within the DLL. cDeviceDriver.cpp, cRegAccess.cpp, and cUsbDetect.cpp provide the mid-level frameworks for SA1350 communication, Windows registry access, and USB detection respectively.

The DLL exports an API which the GUI utilizes from within the Qt application code for interfacing with the hardware. APIs are implemented within sa1350.cpp indicated by SA1350_API at the start of their definitions utilizing the frameworks described above. The available SA1350 DLL APIs are listed and described below.

SA1350 DLL API
  • “sa1350GetDllVersion” - Returns the SA1350 Interface DLL Version
  • “sa1350FindDevices” - Returns the number of detected and installed devices on the current PC
  • “sa1350GetDeviceInfo” - Returns the SA1350 USB Interface Device Information
  • “sa1350Init” - Initializes the SA1350 Interface DLL
  • “sa1350IsInit” - Checks if all the Objects are created and successfully initialized
  • “sa1350DeInit” - Deletes all the generated objects
  • “sa1350GetStatus” - Returns the current device status structure sa1350Status
  • “sa1350Connect” - Connects the dedicated device, given by the comport name
  • “sa1350IsConnected” - Checks if the comport was successfully opened
  • “sa1350Disconnect” - Disconnects the active comport
  • “sa1350IsFrameAvailable” - Checks if the comport has received new frames from the SA1350 device in the meantime
  • sa1350GetFrameFifoSize" - Returns the actual number of received frames within the Frame Fifo Buffer
  • sa1350GetFrame" - Returns the topmost frame from the Frame Fifo Buffer
  • sa1350SendCmd" - Sends a Frame to the SA1350 Device

Qt GUI

The Qt GUI is created with Qt for Application Development using Qt Widgets for most UI elements, the Qt Widgets for Technical Applications (Qwt) library for the plot and plot elements, and imperative C++ for application functionality.

SA1350 GUI Main Window

The main window of the SA1350 GUI is implemented in the MainWindow class which utilizes a QMainWindow as its base class. The class is separated into the standard Qt Widget application files which are a form file, header file, and source file. Inputs from the user are first processed by MainWindow before either being handed off to separate classes which implement more specific functionality or directly handled and responded to by MainWindow itself. Updates to the main window are also processed by MainWindow from either other classes sending information to MainWindow or MainWindow directly responding to previous user interaction directly.

mainwindow.ui is the form file for the MainWindow class and defines the SA1350 GUI’s screen appearance, layout, and widgets which make up the user interface. All user interaction is done to the form, with signals, or events, being generated from the QWidgets inside the form file when the user provides input. Updates to the GUI based upon user input or information updates are done through the form’s QWidgets slots, or handlers.

Both the header and source file of the MainWindow class provide the underlying functionality for the SA150 GUI. The primary responsibilities of mainwindow.cpp and mainwindow.h are to initialize the GUI, implement the event signal processing for GUI inputs, and call the event slot handlers for GUI outputs. Initialization occurs in the class constructor which makes helper function calls to set all GUI elements to initial values, connect event signals to handler functions or event slots, and prepares the GUI for user interaction. All event signal handler functions begin with ‘event’, and all helper functions which make calls to an event slot begin with ‘gui’.

Settings Class

appSettings class is implemented in appSettings.cpp and appSettinsg.h and provides settings save and restore functionality for the SA1350 GUI. The settings constructor creates a new QSettings instance with XML file format and creates or updates an ‘.xml’ file with the current GUI settings. Save and load functionality is available for both GUI and frequency settings.

SA1350 Spectrum Plot

QwtPlot acts as the base class for the spectrum plot of the SA1350 GUI. Spectrum results are displayed or updated in the GUI window upon reception of a complete RF sweep. The results are superimposed on a grid using multi-color traces, with each trace source being distinct and up to four traces selectable. Three markers can also be added to the plot at specific frequency points and for a specific trace. Certain components of the plot are adjustable and the overall plot can be captured for sharing or future reference.

Plot Class

appPlot class is implemented in appPlot.cpp and appPlot.h and provides the primary functionality of the spectrum plot, utilizing other helper classes for specific plot components. The plot constructor accepts an instance of QwtPlot and initializes the components of the plot to default values, turns both the three markers and the four traces off, and prepares data variables for spectrum data. appPlot contains instances of appGrid, appCurve, and appMaker for their respective plot components.

When new data is received by the application, MainWindow provides appPlot with the data. appPlot then processes the data for use by traces then updates active traces with the new data. If new spectrum parameters are set, MainWindow provides appPlot with the new parameters. The frequency and amplitude of the plot are updated as needed, and appPlot prepares for new spectrum data to be received.

Grid adjustments, title changes, and printing or exporting of the plot are also handled by appPlot. Printing is done through the OS print dialog, and the supported export formats include ‘.svg’ and ‘.pdf’.

Grid Class

appGrid class is implemented in appGrid.cpp and appGird.h and provides the grid for the spectrum plot. The grid constructor accepts the QwtPlot instance from appPlot and creates a new QwtPlotGrid instance to provide all supported grid functionality. Grid options are limited to dotted lines, solids lines, or no lines (blank grid) controlled by appPlot.

Curve Class

appCurve class is implemented in appCurve.cpp and appCurve.h and provides the spectrum data line graphs for each active trace. The curve constructor accepts the QwtPlot instance from appPlot, a label, and a color which are stored and used to create a new QwtPlotCurve instance to provide all curve functionality. Curves can be turned on and off, data updated, and color changed by appPlot.

Marker Class

appMarker class is implemented in appMarker.cpp and appMarker.h and provides the markers which can be applied to individual traces at a specific spectrum frequency. The marker constructor accepts the QwtPlot instance from appPlot, a marker number, a color, and a text offset which are stored and used to create a new QwtPlotMarker, QwtSymbol, and QwtText which comprise a marker for the spectrum plot. The marker can be turned on and off, position and data updated, and color changed by appPlot.

Report CSV Class

appReportCsv class is implemented in appReportCsv.cpp and appReportCsv.h and provides a mechanism to output trace spectrum data for analysis. The class consists of a single public function which accepts a filename, spectrum parameter values, frequency values, and corresponding spectrum indication values which are written to a CSV file. The CSV file includes a header for spectrum details and a simple table with frequency and spectrum indication pairs. Trace data export is initiated by MainWindow based upon user input.

SA1350 GUI Device Control

Control of the device connected to the SA1350 GUI is done through the ‘drvSA1350’ driver class which utilizes the DLL’s APIs for device communication and control, a simple state machine for handling device and spectrum events, and a set of commands for requesting or receiving information from the connected device. Connected device status and device spectrum control for both EasyRF and ExpertRF modes is provided by the device control.

SA1350 Driver Class

drvSA1350 class is implemented in drvSA1350.cpp and drvSA1350.h and utilizes QThread as a base class and provides the state machine, application-level device control, and device-level spectrum control. The drvSA1350 constructor initializes the DLL and class variables then starts the driver task (itself) at which point the ‘run’ function executes beginning the driver state machine.

The driver state machine has five states which are described below. All functions related to the state machine begin with ‘state’.

drvSA1350 State Machine States

  • “STATE_INIT” - Confirms DLL initialization, indicates error to user if initialization failed otherwise updates state machine to STATE_OPEN.
  • “STATE_OPEN” - Waits for a USB device connection event, then updates state machine to STATE_SETUP. This also acts as the default state when a currently connected USB device disconnection event occurs or no USB device is connected to the host machine.
  • “STATE_SETUP” - Negotiates with the connected USB device and confirms compliant firmware and device version, then receives RF parameters of the connected USB device and updates the state machine to STATE_RUN.
  • “STATE_RUN” - Either updates the connected SA1350 device’s spectrum parameters when requested by MainWindow or receives the latest spectrum data from the connected SA1350 device.
  • “STATE_EXIT” - Exits the driver task.

All device discovery, information, and connection handling is done with functions which begin with ‘device’. Functions related to spectrum parameters or data all begin with ‘spec’ or ‘spectrum’. Finally, all command functions begin with ‘cmd’ while related helper functions begin with an underscore. A complete list of the supported commands and details of their usage is covered in the Firmware documentation.

The function responsible for all spectrum parameter control for both EasyRf and ExpertRF modes is ‘cmdSetFrq’ called by MainWindow dependent upon user interaction. ‘cmdSetFrq’ sets the connected SA1350 device’s frequency range, receiver bandwidth (RBW), frequency step width (FSW), spectrum’s start and stop frequency, spectrum’s sample count, and spectrum’s span level as both an index into the EasyRF span table and value. ‘cmdSetFrq’ makes use of a set of specific ‘cmd’ functions to perform communication with the SA1350 device.

EasyRF Mode

When operating in EasyRF mode, the user is able to select the frequency range or span to be analyzed from a set of predefined levels. In EasyRF mode the FSW and the RBW of the measurements are automatically adjusted based on the frequency span level selected. The total number of points per sweep in this mode is fixed at 288. The table below describes the predefined spans and the corresponding FSW and RBW values.

Zoom level Span (MHz) Steps/MHz FSW (kHz) RBW (kHz)
0 24 12 83 196 (DEFAULT)
1 18 16 63 155
2 16 18 56 118
3 12 24 42 78
4 8 36 28 59
5 6 48 21 49
6 4 72 14 39
7 2 144 7 39
8 1 288 3 39

At application start, the SA1350 GUI operates in EasyRF mode at zoom level 0, with a center frequency of 915MHz.

ExpertRF Mode

In ExpertRF mode, the user is able to set the frequency span, start, and stop values to any arbitrary value within the supported range. ExpertRF mode also allows the user to independently adjust the FSW and the RBW of the measurements. The number of points per sweep is variable and the maximum number of points is increased from 288 to 2048 points.

Start and stop frequency values are represented as floating point values within the GUI and converted to a fixed point representation matching the SA1350 device architecture prior to sending a frequency command. The SA1350 device expects frequency values with 16-bits for the decimal component of the frequency and 16-bits for the fractional component of the frequency after being multiplied by 65536. See the SA1350 device Technical Reference Manual for more architecture details.

The FSW value is calculated by offsets, or additions, being applied to the SA1350 device architecture specific representation of a fractional frequency value. See notes on architecture in the paragraph above on start and stop frequency values. The minimum supported FSW in ExpertRF mode is 15Hz with a maximum supported value of 1MHz.

Each SA3150 device has a table of supported RBW values, with the RBW value sent to the SA1350 device acting as an index into this RBW table. The RBW table is read from the device as part of the device connection which allows the use of a known index to set the RBW. The SA1350 device’s intermediate frequency (IF) corresponding to the selected RBW is also provided within the table of supported RBW values and shown in ExpertRF mode. See the SA1350 device Technical Reference Manual for more radio frequency details.

For each set of sweep parameters, the number of points is calculated based on the current span and FSW and is used to confirm the correct number of samples has been received from the SA1350 device for each set of new sweep data.

The set of controllable values available in ExpertRF mode give the user complete control over the SA1350 device’s RF parameters.

Status Bar Class

appStatusBar class is implemented in appStatusbar.cpp and appStatusbar.h and provides public functions for updating the status bar of MainWindow. The status bar constructor accepts the QStatusBar instance from MainWindow, starts a periodic timer, and adds the various components displayed within the status bar. Current time, device connection status, DLL and GUI versions, and branding are all displayed within the status bar.

SA1350 Firmware Updater

appFwUpdater class is implemented in appFwUpdater.cpp and appFwUpdater.h with QDialog as a base class and provides a user dialog and mechanism for updating the firmware on the SA1350 Device. The firmware updater construction accepts the parent widget as a parameter, and creates a new QProcess to control the firmware update mechanism. The QProcess events are connected to local event handlers for when the firmware update process is initiated by MainWindow.

MainWindow calls appFwUpdater’s ‘SetParameter’ function to call a script to either install device drivers or start the firmware update process of the SA1350 device. appFwUpdater’s QProcess will set its working directory and script to be executed from MainWindow’s input, then will begin to monitor the progress of the script and update the user with progress within appFwUpdater’s QDialog.

The scripts for both device driver installation and firmware updates are generated using Texas Instrument’s ‘UniFlash’ tool and its Standalone Command Line Package generator.
